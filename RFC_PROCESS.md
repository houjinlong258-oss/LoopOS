# RFC Process

An RFC is required for breaking public schemas, storage migrations, safety-level changes, new
side-effecting syscalls, plugin permission changes, or core dependency additions.

An RFC describes the problem, goals, non-goals, public contracts, security effects, compatibility,
alternatives, test plan, and rollback. It progresses through draft, discussion, accepted or rejected,
implementation, and complete states.

At least one core maintainer and one independent reviewer must approve an RFC. Security-sensitive
details may be reviewed privately before a public summary is published.
